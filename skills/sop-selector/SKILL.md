# sop-selector：SOP 选择框架

## 概述

本 Skill 是一个**参考型（reference）** Skill，直接注入调用方 Agent 的上下文。
Manager 扫描 `shared/sop/` 目录，按下面的框架选出最匹配的 SOP，
写入 `shared/sop/active_sop.md` 作为本次任务的执行规范。

---

## 三步选择框架

### 第一步：分析需求特征

从 `requirements.md` 中提取以下特征：
- **任务类型**：产品设计 / 技术开发 / 代码评审 / 测试 / 其他
- **规模与复杂度**：功能数量 / 涉及角色数 / 是否跨系统
- **时间约束**：是否有明确 deadline，是否是紧急交付
- **待确认项数量**：需求中「待确认」标注越多，越需要完整流程

### 第二步：候选 SOP 评分

对 `shared/sop/` 目录下每个 SOP（排除 `draft_` 前缀和 `active_sop.md`）：

| 维度 | 评分依据 |
|------|---------|
| 任务类型匹配度 | SOP 适用场景 vs 当前需求类型 |
| 规模匹配度 | SOP 步骤粒度 vs 需求复杂度 |
| Checkpoint 覆盖 | SOP 中 Checkpoint 数量是否合理 |

**特殊规则**：
- 如果 SOP 库中只有一个文件，**直接选择该文件**，无需评分比较，节省时间。
- 如果所有 SOP 匹配度都低于 60%，选分数最高的并在推荐理由中注明缺口。

### 第三步：输出推荐

输出以下内容：
1. **选中的 SOP**：文件名 + 一句话说明为什么选它
2. **注意事项**（如有）：当前任务有哪些地方需要在执行时特别注意

---

## 输出约定

选中 SOP 后，调用 `memory-save` skill，将选中 SOP 的完整内容写入：
- 路径：`/mnt/shared/sop/active_sop.md`
- 写入方式：**始终覆盖写入**（action=write），不检查文件是否已存在

`active_sop.md` 是本次任务的 SOP 副本，内容与源文件相同，但与源文件解耦：
即使后续 SOP 库更新，本次任务的执行规范不受影响。

---

## 使用注意

- 本 Skill 是**纯思考框架**，不执行任何代码
- 扫描 SOP 目录时，忽略文件名以 `draft_` 开头的草稿文件
- 忽略 `active_sop.md`（这是上次任务的副本，不是 SOP 模板）
- 选择结果以简洁推荐理由为主，不需要完整的评分报告（人只需知道「选了哪个，为什么」）
