# 产品设计精简 SOP（Lite 版）

## 适用场景

需求清晰、范围小、时间紧的快速交付场景：
- 单功能改造（非全新系统）
- 工期约束在 1 周以内
- 验收标准已在需求文档中明确，无需进一步拆解

不适用：多模块协作、跨团队对接、存在大量待确认项的复杂需求。
这类场景请使用完整版 SOP（product_design_sop.md）。

## 角色分工

| 角色 | 职责 |
|------|------|
| Manager | 确认需求范围、分配任务、验收结论 |
| PM | 直接输出产品规格，跳过拆解阶段 |

## 执行步骤（2步）

### 第一步：需求确认 + 任务分配（Manager）

1. 读取 `/mnt/shared/needs/requirements.md`
2. 确认以下内容已完整：目标 / 边界 / 验收标准
3. 通过 mailbox-ops 向 PM 发送任务：
   - `type: task_assign`
   - `content`: 直接列出核心功能点和验收标准，无需另写设计说明

### 第二步：产品规格输出（PM）

产品规格文档路径：`/mnt/shared/design/product_spec.md`

精简格式（必须包含）：

```markdown
# 产品规格（精简版）

## 功能说明
[一句话描述核心功能]

## 接受标准
- [ ] [可测试的验收条件1]
- [ ] [可测试的验收条件2]

## 接口约定（如有）
[关键接口或数据格式说明]
```

文档完成后，通过 mailbox-ops 向 Manager 发送完成通知：
- `type: task_done`
- `subject`: 产品文档（精简版）已完成

## Checkpoint

| 位置 | 类型 | 说明 |
|------|------|------|
| PM 完成产品规格后 | checkpoint_request | Manager 转发给人类确认，确认通过后进入开发 |

## 质量要求

- 文档必须可执行、可测试
- 验收标准必须量化，不得使用「用户体验更好」等模糊表述
- 文档字数不设下限，以「能指导开发」为准
